# Design: GGA Pre-commit Integration

## Technical Approach

Two static config files (`.gga`, `.gga-rules.md`) plus a hook already installed during exploration (`gga run || exit 1`). GGA reads the rules file on every commit, builds a prompt with staged Go files, reviews via opencode, and blocks the commit on FAIL.

## Architecture Decisions

### Decision: Standalone Rules File

| Option | Tradeoff | Decision |
|--------|----------|----------|
| `.gga-rules.md` (standalone) | Decoupled from AGENTS.md. Rules stable across skill-sync regenerations. | ✅ **Chosen** |
| Reference `AGENTS.md` | Single source of truth. But AGENTS.md is auto-generated by skill-sync — GGA rules would be lost on regeneration. | ❌ Rejected |

### Decision: Bash-sourced Config Format

GGA's runtime sources `.gga` as shell `KEY="VALUE"` pairs. Follows the same format as the global `~/.config/gga/config`. No parser needed — GGA consumes it natively.

### Decision: STRICT_MODE=true

When the AI response is ambiguous (no explicit PASS/FAIL), strict mode fails the review. Safer default: false negatives (passing bad code) cost more than false positives (blocking good code that can `--no-verify`).

## Data Flow

```
git commit
  → .git/hooks/pre-commit (already installed)
    → gga run
      → source .gga → PROVIDER, RULES_FILE, FILE_PATTERNS
      → read .gga-rules.md → rules prompt
      → git diff --cached --name-only → staged file list
      → filter by FILE_PATTERNS (*.go,*.md,*.mod,*.sum)
      → build AI prompt: rules + file contents + diff
      → call opencode provider
      → parse response for STATUS: PASS | FAIL | WARN
      → PASS → exit 0 (commit proceeds)
      → FAIL → print feedback → exit 1 (commit blocked)
```

## File Changes

| File | Action | Description |
|------|--------|-------------|
| `.gga` | Create | GGA config: provider, model, rules file, patterns, timeout |
| `.gga-rules.md` | Create | AI-consumable coding standards (Go + Bubbletea + project) |
| `.git/hooks/pre-commit` | Already exists | `gga run \|\| exit 1` — no change needed |

## File Designs

### `.gga`

```bash
# GGA Configuration — skillsync/TUI
PROVIDER="opencode"
MODEL="opencode/deepseek-v4-flash-free"
RULES_FILE=".gga-rules.md"
FILE_PATTERNS="*.go,*.md,*.mod,*.sum"
EXCLUDE_PATTERNS="*_test.go"
STRICT_MODE="true"
TIMEOUT="300"
```

### `.gga-rules.md` — Structure

Sections (each 3-6 bullet rules):

**Go Conventions**: `gofumpt` compliance, PascalCase exports / camelCase privates, acronyms as `ID`/`HTTP`, `if err != nil` checked immediately, `fmt.Errorf("ctx: %w", err)` wrapping, no panics in lib packages.

**Bubbletea MVU**: `Model`-`Update`-`View` separation, `tea.Cmd` for all side effects (never inline I/O in Update), typed `tea.Msg` structs for events, `Screen` enum for navigation, `tea.Quit` on `ctrl+c`, sub-model composition.

**Project Conventions**: Import order (stdlib → external → `skillsync/tui/internal`), `AppService` interface for service abstraction, `0644`/`0755` permissions, no `log.Fatal` outside `main()`, table-driven tests with `t.Parallel()`.

**Review Rules**: PASS = all rules followed, FAIL = any MUST violation, WARN = SHOULD violation. Security: no hardcoded creds, no command injection via `os/exec` with unsanitized input. Resource: check file handle leaks, goroutine lifecycle.

## Testing Strategy

| Layer | What | How |
|-------|------|-----|
| Unit | `.gga` syntax | Source in bash, verify vars set |
| Integration | `gga run` on staged file | Stage a valid Go file, run `gga run`, expect PASS |
| E2E | Hook fires on commit | `git commit` with clean code → hook runs → commit succeeds |
| E2E | Hook blocks bad code | `git commit` with violating code → hook fails → commit blocked |

## Migration / Rollout

No migration required. Both files are new. Hook already installed from exploration.

## Rollback

1. `git rm .gga .gga-rules.md`
2. `gga uninstall` (removes `.git/hooks/pre-commit`)
3. Commit the removal

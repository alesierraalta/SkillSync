## Review

- Correct: README and provider picker agree on the documented Gemini 3 Flash id. README lists `gemini-3-flash-preview` in the chat model table (`README.md:25-31`), and the picker source registers the same id in `GEMINI_MODEL_CHOICES` (`src/config/model.ts:43-47`).
- Correct: tool calls do pass the configured model to the Gemini CLI process. `buildGeminiAcpCommandSettings()` appends `--model <settings.model>` when the user's args do not already include a model flag (`src/acp/settings.ts:4-18`), and both prompt and search paths build command settings from provider settings before calling ACP (`src/prompt/run.ts:155-158`, `src/search/run.ts:122-127`).

- Blocker: `/gemini-model gemini-3-flash` can persist a syntactically valid but nonexistent model id, reproducing the reported `Requested entity was not found` failure. Evidence: `resolveGeminiModelName()` validates arbitrary full-looking Gemini ids before checking known choices/aliases (`src/config/model.ts:212-220`), and the regex accepts `gemini-3-flash` (`src/config/model.ts:20`). The later probe only checks whether command help advertises `--model`, not whether the selected model exists (`src/config/model.ts:234-241`). Because `buildGeminiAcpCommandSettings()` blindly appends the persisted setting (`src/acp/settings.ts:7-8`), `gemini_ask`/search start the CLI with `--model gemini-3-flash` and fail until settings are changed to `gemini-3-flash-preview`.
  - Recommended fix: prefer exact known choices/aliases before arbitrary regex acceptance, and add a compatibility alias/migration for the known bad value `gemini-3-flash` -> `gemini-3-flash-preview` (similarly consider `gemini-3-pro` -> `gemini-3-pro-preview`). For direct arbitrary model ids, consider warning that the id is unverified or doing a lightweight model-existence probe when possible.

- Blocker: Pi chat model picker selection does not appear to control the actual Gemini CLI model. `buildGeminiAcpProviderConfig()` creates one cached prompt client from the persisted provider settings before registering all picker models (`src/models/provider.ts:55-61`), and `createGeminiAcpStreamSimple()` receives only that prebuilt client plus chat config; at stream time it uses `model.id` only for preamble/metadata/usage, then calls `client.prompt(request, ...)` without applying `model.id` to command settings (`src/models/stream.ts:130-192`). Result: selecting `gemini-3-flash-preview` in Pi chat can still run whatever is in `providers['gemini-acp'].model`/args, including the bad `gemini-3-flash` value. This matches the reported mismatch between picker/README and runtime failure.
  - Recommended fix: make `streamSimple` create or fetch a client keyed by the Pi-selected `model.id` (e.g. append/override `--model model.id` for the chat request), or register only a single model that reflects the persisted CLI model. If Pi picker models are meant to be independent from `/gemini-model`, the runtime must honor the selected model id.

- Note: `settings.args` can silently override `settings.model`. `hasModelArg()` suppresses appending the persisted model if args already include `--model`, `-m`, or `--model=...` (`src/acp/settings.ts:16-18`), but `geminiAcpModelLabel()` prefers `settings.model` over `modelFromArgs()` for UI/cache/fallback labeling (`src/config/model-label.ts:10-14`). If args contain one model and settings.model contains another, the CLI runs one model while progress, quota cache, API fallback, and cost metadata report/use another.
  - Recommended fix: either make configured args model take precedence everywhere, reject conflicting model sources, or normalize model configuration into one source of truth.

- Note: Search's `request.model` is display/progress context, not model selection. `runSearch()` passes `model` into `client.search()` (`src/search/run.ts:170-175`), but ACP client selection is already fixed by `commandSettings`; the search prompt builder ignores `request.model` (`src/acp/search-prompt.ts:4-5`). This is fine if command settings are authoritative, but it reinforces that fixing command-settings/model selection is required for production correctness.

- Note: I could not verify tests locally. `npm test -- --runInBand` failed because `vitest` is not available on PATH in this installed package (`'vitest' is not recognized...`). The package also appears to ship no repo-local `src/**/*.test.ts` or `src/**/__tests__/*.ts` files in this node_modules copy, despite test scripts referencing Vitest.

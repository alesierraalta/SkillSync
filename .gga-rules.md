# GGA Review Rules — skillsync/TUI

This file is prepended to every GGA review prompt. The AI reviewer MUST follow these standards when evaluating code changes.

---

## Section 1: Go Language Standards

### Formatting
- Code MUST pass `gofmt` or `gofumpt` — no tabs vs spaces debate, gofmt rules are final.
- Lines should be wrapped at a reasonable column (default 120). Do not use hard line wraps on prose strings.
- Group consecutive single-import blocks into one parenthesized `import` block.

### Naming
- **Exports**: PascalCase — `type SyncReport struct`, `func Parse(path string) (*Skill, error)`
- **Privates**: camelCase — `func slugifySkillName(prompt string) string`, `var supportedRoots`
- **Avoid stutter**: Package name + type should not repeat. Use `coreskills.Config`, NOT `coreskills.CoreSkillsConfig`. Use `runner.SyncReport`, not `runner.RunnerSyncReport`.
- **Acronyms**: All caps or all lower — `HTTP`, `URL`, `ID`. Not `Http`, `Url`, `Id`.
- **Interfaces**: Name by behavior, suffix with `-er` where appropriate. Example: `AppService` (not `IAppService` or `AppServiceInterface`).

### Error Handling
- Check `if err != nil` immediately after the call that produced the error. No empty lines between the call and the check.
- Wrap errors with `fmt.Errorf("context: %w", err)` using `%w` for error chain propagation.
- Use `errors.New` only for simple sentinel errors that callers need to check with `errors.Is`.
- NEVER use `_ =` to discard errors unless there is an explicit comment explaining why the error is intentionally ignored.
- Example — GOOD:
  ```go
  f, err := os.Open(path)
  if err != nil {
      return fmt.Errorf("open config: %w", err)
  }
  ```
- Example — NON-FATAL DISCARD (must be documented):
  ```go
  _ = m.backend.RegisterProject(absRoot) // non-fatal; registration is best-effort
  ```

### Panics
- **NEVER** use panics in library or internal packages. They are reserved for truly unrecoverable states in `main()` only.
- Even in `main()`, prefer `fmt.Fprintf(os.Stderr, "...")` + `os.Exit(1)` over `panic()`.
- Flag.Parse errors should be returned, not panicked.

### Testing
- Use **table-driven tests** with `t.Run(tt.name, ...)` for all multi-case scenarios. See `TestNewCLICommands` and `TestFindProjectRoot_MultiProvider` in `cmd/synck/main_test.go`.
- Call `t.Parallel()` at the top of each test function and each sub-test when tests are independent.
- Use descriptive test names that document the scenario. Example: `TestHandleSync_VerboseFlag`, not `TestSync3`.
- Prefer assertion helpers over raw `if` blocks when the same check repeats. Example helper:
  ```go
  func assertFileExists(t *testing.T, path string) {
      t.Helper()
      if _, err := os.Stat(path); os.IsNotExist(err) {
          t.Errorf("expected file to exist: %s", path)
      }
  }
  ```
- Use `t.TempDir()` for temp directories — they auto-clean.
- For stdout capture tests, use pipes with goroutines to avoid deadlocks. See `TestHandleSync_ProgressAndReport`.
- Avoid mocking where possible. Prefer pure function extraction for testability.

## Section 2: Bubbletea MVU Standards

### Model
- The `Model` struct is the single source of truth for UI state.
- Fields include: current `Screen`, `Width`/`Height` for layout, `StatusMsg`, sub-models (e.g., `Installer`, `List`, `deleteConfirm`), and service layer (`backend AppService`).
- Screen constants use `iota`:
  ```go
  const (
      ScreenHome Screen = iota
      ScreenList
      ScreenDetail
      ...
  )
  ```
- Initialization via `NewModel(...)` constructor — never zero-initialize a Model directly.

### Update
- **Handle ALL possible messages** in the `Update` method's type switch. Unhandled messages must fall through to a catch-all.
- Signature MUST be `func (m Model) Update(msg tea.Msg) (tea.Model, tea.Cmd)`.
- Use `tea.Sequence` for ordered commands where execution order matters.
- Use `tea.Batch` for parallel commands.
- Delegate to screen-specific handlers: `handleHomeKeys`, `handleListKeys`, etc.
- Delegate to component updates: `m.List.Update(msg)`, `m.Installer.Update(msg)`.
- Custom messages are typed structs implementing `tea.Msg`:
  ```go
  type syncReportMsg struct {
      report *runner.SyncReport
      err    error
  }
  ```

### View
- **Pure rendering only** — no side effects, no mutations, no I/O.
- Use `lipgloss` for all styling. Define reusable styles as package-level `var` blocks.
- Compose styles with `.Render(s)` and string concatenation.
- Screen dispatch pattern:
  ```go
  func (m Model) View() string {
      switch m.Screen {
      case ScreenHome:
          return m.homeView()
      case ScreenList:
          return m.listView()
      ...
      }
  }
  ```

### Commands
- All I/O (file reads, process execution, network, timers) MUST be wrapped in `tea.Cmd`.
- NEVER block in `Update` or `View` — blocking hangs the TUI event loop.
- `tea.Cmd` signature: `func() tea.Msg` — wrap side effects in closures.
- Pattern for async operations:
  ```go
  func (m Model) loadSkills() tea.Cmd {
      return func() tea.Msg {
          paths, err := m.backend.DiscoverSkills(m.rootPath)
          if err != nil {
              return errorMsg(err)
          }
          return skillsLoadedMsg(skills)
      }
  }
  ```

### Messages
- Custom messages are exported struct types in the `ui` package.
- Type switch on `msg.(type)` in `Update` dispatches to the right handler.
- Use descriptive names: `syncFinishedMsg`, `skillsLoadedMsg`, `installerProgressMsg`.
- Embed errors in the message struct rather than sending separate error messages:
  ```go
  type syncFinishedMsg struct { err error }
  ```

### Component Composition
- Sub-models (e.g., `ListModel`, `InstallerModel`, `DeleteConfirmModel`) have their own `Update` and `View` methods.
- Parent model delegates by calling `m.Component.Update(msg)` and casting the result.
- Sub-models are initialized in the parent's constructor (`NewModel`).
- `KeyBinding` type documents available keys:
  ```go
  type KeyBinding struct {
      Key  string
      Help string
  }
  ```

## Section 3: Project Conventions

### Import Order
Imports MUST be grouped in this order with blank lines between groups:

1. **stdlib** — standard library packages
2. **external dependencies** — third-party packages (e.g., `github.com/charmbracelet/bubbletea`)
3. **internal packages** — `skillsync/tui/internal/...`

```go
import (
    "fmt"
    "os"
    "path/filepath"

    tea "github.com/charmbracelet/bubbletea"
    "github.com/charmbracelet/lipgloss"

    "skillsync/tui/internal/runner"
    "skillsync/tui/internal/storage"
    "skillsync/tui/internal/types"
)
```

### Handler / Service Pattern
- Use **interface types** for service layer dependencies (e.g., `AppService`).
- Receive dependencies via constructor injection (`NewModel(backend AppService)`, `NewBackend(storageService *storage.Service)`).
- Backend implementations wrap concrete packages (`discovery.DiscoverSkills`, `parser.Parse`, etc.).
- Package-level convenience functions may instantiate a default backend when callers don't need injection.

### File Permissions
- **0644** for all created files (`os.WriteFile(..., 0644)`)
- **0755** for directories and executable scripts (`os.MkdirAll(..., 0755)`)

### Error Patterns
- `fmt.Errorf("context: %w", err)` for all wrapped errors.
- `errors.New` only for sentinel errors that must be checked with `errors.Is`.
- Temporary/staging error variables are `return`ed, never `log.Fatal`'d.
- Best-effort/should-not-block operations capture the error but don't fail the operation:
  ```go
  if _, err := opencode.SyncSkills(".", opencode.Options{}); err != nil {
      // Non-fatal: continue even if sync fails
  }
  ```

### Concurrency
- Goroutine lifecycle MUST be documented with a comment explaining when/why it exits.
- Use `sync.WaitGroup` or `errgroup.Group` for goroutine cleanup.
- Context cancellation MUST be propagated through all goroutines.
- Channel types should clearly state direction (`chan<- Msg` for send-only, `<-chan Msg` for receive-only) where appropriate.

### Configuration
- Use environment-based or file-based configuration. Never hardcode paths.
- Viper may be used when configuration is complex enough to warrant it.
- Default to `.` (current directory) as the root path, with overrides configurable.

### String Constants and Version
- Version strings and constants are package-level variables, not magic strings inline.
- Example: `const version = "0.1.0"`, `var supportedRoots = []string{...}`.

## Section 4: Review Rubric

### PASS
Code follows ALL standards in Sections 1–3. No issues found. No warnings raised.

### FAIL (Critical — block commit)
Reviewer MUST return FAIL if any of the following are detected:

- **Security issue**: Hardcoded credentials, API keys, tokens, or secrets in source code.
- **Command injection**: User input passed to `os/exec` without sanitization, or shell-escaped arguments via `sh -c` with unchecked strings.
- **Path traversal**: User-controlled input used in file paths without validation or `filepath.Clean`.
- **Panic in library code**: `panic()` or `log.Fatal()` called in any `internal/` package.
- **Data race**: Shared mutable state accessed from multiple goroutines without synchronization (mutex, channel, or atomic).
- **Resource leak**: File handle, HTTP response body, or database connection opened but not closed via `defer`.
- **Unchecked error in critical path**: Error returned from an I/O operation or resource allocation is silently discarded without comment.

**FAIL verdict format**:
```
FAIL: [Category] — [File]:[Line] — [Description]
```

### WARN (Style or minor — does not block commit)
Reviewer SHOULD return WARN for any of the following:

- Import group order does not follow Section 3 conventions.
- Missing `t.Helper()` call in test assertion helper function.
- Error message uses `%s` instead of `%w` for error wrapping.
- Unused function parameter (use `_` to silence the warning).
- Inconsistent naming with project conventions (e.g., camelCase when PascalCase is expected).
- Comment explains WHAT instead of WHY (comments should explain intent, not describe the code).
- Cyclomatic complexity in a single function exceeds 15 (consider extraction).
- Magic number or string that should be a named constant.
- No `t.Parallel()` in a table-driven test with independent cases.

**WARN verdict format**:
```
WARN: [Category] — [File]:[Line] — [Description]
```

### Security Focus Areas

#### Credentials
- NO credentials, tokens, API keys, or secrets in source code.
- NO .env files committed to version control.
- NO connection strings with embedded passwords.
- Use environment variables or a secrets manager for all sensitive values.

#### Command Injection Prevention
- Use `os/exec` with separate arguments (`cmd.Args`), NEVER shell-quoted strings.
- Avoid `sh -c "..."` with user input unless input is strictly validated against a whitelist.
- Example — SAFE:
  ```go
  cmd := exec.Command("git", "log", "--oneline", "-5")
  ```
- Example — UNSAFE (block by FAIL):
  ```go
  cmd := exec.Command("sh", "-c", fmt.Sprintf("git log %s", userInput))
  ```

#### Path Traversal Prevention
- Use `filepath.Join` for constructing paths, never string concatenation.
- Use `filepath.Clean` and `filepath.Abs` to resolve and validate user-supplied paths.
- Check that resolved paths stay within an expected base directory.

### Resource Discipline
- Every `os.Open`, `os.Create`, `http.Get`, `db.Query` must have a corresponding `Close` or `defer Close`.
- Every goroutine launched must have a documented shutdown path (context cancellation, channel close, or WaitGroup).
- Context cancellation MUST be propagated: if a function accepts `context.Context`, it must respect cancellation in blocking operations.
- Temporary files created in tests MUST use `t.TempDir()` or be explicitly cleaned up.

## AI Reviewer Instructions

1. Review ALL staged files matching `FILE_PATTERNS` (excluding `EXCLUDE_PATTERNS`).
2. Apply the rubric above: return the first FAIL found (fail fast), or list all WARNs.
3. For each finding, cite the exact file path and line number.
4. Include a brief code snippet for context where helpful.
5. CRITICAL: Your response MUST begin with a `STATUS:` line on the VERY FIRST LINE. Use one of:
   - `STATUS: PASSED` — no FAIL findings (WARNs are allowed and do not block the commit)
   - `STATUS: FAILED` — at least one FAIL finding was detected
6. Format the review as:

```
STATUS: PASSED

## Review Result: {PASS|WARN}

### Findings
{Warn items with file:line references, or "None" if clean}

### Summary
{Overall assessment — 2-3 sentences}

### Suggestion (if WARN)
{Optional improvement suggestion if not a blocker}
```

Or for failures:

```
STATUS: FAILED

## Review Result: FAIL

### Findings
FAIL: [Category] — [File]:[Line] — [Description]

### Summary
{Overall assessment}
```

The `STATUS:` line MUST be the absolute first line of your response. No preamble, no headers before it.
